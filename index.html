递归(recursion)

Posted on 2019-08-01

递归的定义

递归是一种数学思想
递归函数的定义：一个直接调用自己或通过一系列的调用语句间接地调用自己的函数

递归的作用

递归是程序设计中一个强有力的工具，主要体现在三个方面

  1. 有很多的数学函数是递归定义的
  • 阶乘函数

  • 2阶Fibonacci数列（斐波那契数列）

  • Ackerman 函数

  1. 有的数据结构，由于结构本身固有的递归特性，则它们的操作可递归地描述
  • 二叉树
  • 广义表
  1. 还有一类问题，虽然问题本身没有明显的递归结构，但用递归求解比迭代求解更简单
  • 八皇后问题
  • Hanoi塔问题

结论

移动n个圆盘需要2^n-1次

递归的应用

递归在编程语言中的实际应用
汇编主程序和子程序之间的链接及信息交换类似，在高级语言编制的程序中，调用函数和被调用函数之间的链接及信息交换需通过调用栈来进行。

这里有几个知识点：

  1. 函数调用方式：栈
  2. 栈的数据结构和操作
  3. 递归函数的调用，实际就是相同函数的调用，c语言是允许递归调用的

Telegram_iOS Swift项目架构

Posted on 2019-07-30 Edited on 2019-08-02

源码分析-项目架构

好了顺利跑起来工程后，我开始分析它的源码，Swift项目比 Objective-C项目要清晰得多，作者都是peter-iakovlev。我们从提交记录就可以大致看出，我们也能从两个项目中感受到作者通过这个项目的进步。

这里我以一张图来简单介绍下项目的架构：

其中，绿色是swift语言，棕黄色是OC语言实现，白色是C和C++实现。从数量上我们大致有个印象。

UI视图层

组件化的思想提现得很明显，其中UI大部分放在了TelegramUI这个工程下面，看了下细节的实现，大部分采用facebook工程是的第三方的UI异步绘制框架Texture（以前叫AsyncDisplayKit）来提高界面的流畅性
而对于异步事件的处理，采用了ReactiveX响应式编程的思想。但是值得注意的一点是，作者用的也不是主流的Rxswift框架，而是类似于这个框架的SSignalKit。这里不知道是出于什么考虑，为什么不用业界的框架，这样对于我们来说，有一定的上手难度，但是思想都差不多啦。

业务层

在业务层，把UI和逻辑处理抽离开来形成了TelegramUI 和 Telegram-iOS这两个组件。Telegram-iOS我们可以理解为这个工程的组织者，他负责组织和协调各个组件。

协议层

在协议层，用了自家非常安全的MTProtoKit协议，已经对TL语言的swift实现TelegramApi，能够自己定义一套数据的表达语言，在多端同步使用，足以见起逼格。关于MTProto协议，我后面会专门花一篇文章来分析介绍。它是在TCP协议的基础上，自定义了一套安全的传输协议。

基础组件层

在基础组件层，作者也是展示了自己的架构能力，把每一个功能模块独立出来，特别需要强调的是TelegramCore这个库，从命名就能感受到他的重要性，作为核心支撑库，后面我们也会来进行详细分析。在基础组件库中，有一个有意思的LegacyComponents库。里面的应该是OC项目遗留下来的，没有来得及重构的组件，这里也可以看到一个软件开发的过程。任何一个厉害的开源项目都是这样持续不断的重构和迭代来的。

三方组件

在第三库上，作者用得还算比较克制，没有过多的用第三方库，都是业界非常著名的开源库。

总结

通过这两天对源码的分析，我收获了很多，也对之前的自己的一些架构有所反思。写文章我们也可以用这样的思想，小步迭代，慢慢完善。

Telegram-iOS Swift 编译问题

Posted on 2019-07-29 Edited on 2019-08-01

前面一篇文章Telegram-iOS github 拉取失败解决过程我们介绍了如何从github上面拉取Telegram-iOS的源码，在文章的结尾我说会介绍下如何能将项目跑起来，这里兑现我的承诺。
刚开始我也是按照官方的文档，拉取代码，注意一定要拉取子模块，即使用
git clone —recursive
命令，这样才能保证项目的完整性，在第五步中，让我们
Open the Telegram-iOS-Fork scheme.
我也按照这个步骤选中了这个scheme.
关于什么是scheme,我想搞iOS开发一定知道这个，可以理解成一个编译计划，编译哪个target,怎么编译，环境变量是哪些。scheme可以选择属于project还是workspace。一个scheme对应一个target,一个target可以对应多个scheme。具体可以看苹果官方文档
废话不多说，当我按照这个步骤拉取代码后，并选中 Telegram-iOS-Fork这个scheme后，设备选中模拟器编译，还是报错了，保持信息如下
No such module ‘SwiftSignalKit’ in TelegramAudio Framework
但是我看了，在 Linked Frameworks and Libraries中已经定义过了。

在github Issues中我找到了同样的问题
Unable to compile · Issue #82 · TelegramMessenger/Telegram-iOS · GitHub
看回答也是说选这个scheme,但是我这边还是报错了，回答中有一说是重复的问题
release-5.8 WatchCommonWatch/WatchCommonWatch.h not found · Issue #76 · TelegramMessenger/Telegram-iOS · GitHub

这里有个回答说用AppStoreLLC scheme 可以编译成功，我就试了下，果然通过了。
比我想象中要顺利得多,至于具体什么原因我还没去深究，应该是工程配置的问题。后面再去查下，有知道的同学欢迎留言。

遇到问题，我们多去看官方的一些Issue,你遇到的可能别人早已经遇到并解决过了。

Core ML

Core ML的API很简单，你所要做的就是：

  1. 加载一个训练过的模型
  2. 生成预测
  3. 发布

在Core ML之前，要加载一个训练过的模型非常麻烦，而现在，加载模型只需要两个步骤。

模型包含在.mlmodel文件里，这是一种新的开放文件格式，它描述了模型的层次，输入和输出、类标签，以及一些必要的预处理步骤。它还包含了所有学习过的预处理参数（权重和偏差）。所有我们需要的东西都包含在这个文件里。

我们之需要把mlmodel文件放在我们项目里，Xcdoe会自动为它生成一个Swift或Objective-C包装类。

Core ML 会自己决定是该用CPU还是GPU来允许模型，这样可以更好的使用资源。甚至，Core ML 还可以将模型拆分开，部分运行在GPU(需要更多计算的任务)上，部分运行在CPU(消耗更多内存的任务)上。

Core ML 能够使用CPU对于开发者来说是一件好事，因为这样它们就可以在iOS模拟器上运行（而Metal是不允许这样做的，所以也就无法进行单元测试）

Core ML支持哪些模型

  • 支持向量机(SVM)
  • tree ensemble (like: random forest/ boosted tree)
  • 线性回归和逻辑回归
  • 神经网络： 前馈、卷积、循环

苹果提供了一系列经过训练的模型进行下载：

  • Inception v3
  • RestNet50
  • VGG16

同时，我们也可以使用Core MLPython工具包将你自己的模型转换成可用的模型。

  • Keras
  • Caffe
  • scikit-learn
  • XGBoost
  • libSVM

Core ML 的局限

  • 它只支持有监管的机器学习模型，不支持无监管的学习算法或者增强型学习。
  • 不能在设备上进行训练，你需要使用离线工具进行训练，然后把训练过的模型转换成Core ML格式。
  • 如果Core ML不支持某种层次类型，那么你就不能使用它。目前为止，使用自己的计算内核是无法对Core ML进行扩展的。TensorFlow可以用于构建一般性的计算图，而mlmodel文件并不像它那么灵活。
  • Core ML转换工具只支持一部分指定版本的训练工具。如果你在TensorFlow里训练模型，就无法使用这个工具，必须自己编写转换脚本。
  • 你无法查看神经网络中间层生成的结果，只能获得最后层的预测。
  • 下载模型更新可能会有问题
  • Core ML 把选择在CPU或者GPU上运行的细节隐藏起来了，你要相信它能做出正确的选择，你无法强制要求它在哪运行。
